# Building a Native Module in Expo — From Bridge to JSI

A step-by-step guide on writing a real Android native module, understanding
why the old bridge approach falls short, and converting it into a proper
JSI-backed Expo Module that survives `expo prebuild` and belongs in git.

---

## What we're building

A **Device Info** module that exposes Android hardware data to JavaScript:

- `getBatteryLevel()` — battery percentage (0–100)
- `isCharging()` — whether the device is plugged in
- `getDeviceInfo()` — model, brand, OS version, SDK, manufacturer
- `constants.DEVICE_NAME / OS_VERSION` — sync values, no async call needed

---

## Part 1 — The Old Way (React Native Bridge)

Before understanding the Expo Module approach, it helps to know what came before it.

### How the classic bridge works

```
JS → JSON serialize → async message queue → Kotlin → JSON serialize back → JS
```

Every call hops across a bridge thread, serialises all data to JSON, and
deserialises it on the other side. For frequent calls this adds real latency.

### Writing a bridge module

You need three things:

**1. The module class**

```kotlin
// android/.../DeviceInfoModule.kt
class DeviceInfoModule(reactContext: ReactApplicationContext) :
    ReactContextBaseJavaModule(reactContext) {

    override fun getName(): String = "DeviceInfo"   // JS looks this up

    @ReactMethod
    fun getBatteryLevel(promise: Promise) {         // Promise is mandatory
        val pct = ...                               // query Android API
        promise.resolve(pct)                        // send value back to JS
    }
}
```

**2. A Package class** to register it with React Native:

```kotlin
class DeviceInfoPackage : ReactPackage {
    override fun createNativeModules(ctx: ReactApplicationContext) =
        listOf(DeviceInfoModule(ctx))

    override fun createViewManagers(ctx: ReactApplicationContext) =
        emptyList<ViewManager<*, *>>()
}
```

**3. Manual registration** in `MainApplication.kt`:

```kotlin
add(DeviceInfoPackage())
```

**JS side** — raw and untyped:

```ts
import { NativeModules } from 'react-native'
const { DeviceInfo } = NativeModules   // no types, no autocomplete
await DeviceInfo.getBatteryLevel()     // works, but fragile
```

### The problem with this approach

1. **It's not JSI** — data goes through JSON serialisation on every call
2. **The `android/` folder is gitignored** in Expo projects because it's
   generated by `expo prebuild`. Your module files live there and get wiped
   whenever someone runs `prebuild --clean`
3. **Three files of boilerplate** just to expose one method

---

## Part 2 — The Expo Module Way

Expo Modules (`expo-modules-core`) solve all three problems:

- Uses **JSI** under the hood — no JSON, no bridge thread, direct C++ calls
- Lives in a **`modules/` folder** that you commit — survives prebuild
- A **config plugin** wires it into `android/` automatically on every prebuild
- **One Kotlin class**, no Package class, no MainApplication change

### Architecture overview

```
modules/device-info/           ← you commit this
  package.json                 ← local npm package declaration
  expo-module.config.json      ← tells autolinking which class to register
  plugin/index.js              ← config plugin: patches android/ on prebuild
  src/index.ts                 ← TypeScript public API
  android/
    build.gradle               ← Android library config
    src/main/
      AndroidManifest.xml
      java/expo/modules/deviceinfo/
        DeviceInfoModule.kt    ← the Expo Module (JSI-backed)
        DeviceInfoRecord.kt    ← typed return value

android/                       ← gitignored, generated by prebuild
  settings.gradle              ← config plugin adds :device-info include here
  app/build.gradle             ← config plugin adds implementation dependency here
```

## Part 3 — Building it step by step

### Step 1 — Create the local package

`modules/device-info/package.json`:

```json
{
  "name": "device-info",
  "version": "1.0.0",
  "private": true,
  "main": "src/index.ts"
}
```

Register it in the root `package.json` so Node can resolve the import:

```json
"dependencies": {
  "device-info": "file:./modules/device-info"
}
```

Add a TypeScript path alias in `tsconfig.json` so the compiler resolves it
without needing `npm install` first:

```json
"paths": {
  "device-info": ["./modules/device-info/src/index.ts"]
}
```

### Step 2 — expo-module.config.json

This is the most important file. It tells `expo-modules-core`'s autolinking
which Kotlin class implements the module. Without it, the JSI function
bindings are never generated and your async functions show up as `undefined`.

```json
{
  "platforms": ["android"],
  "android": {
    "modules": ["expo.modules.deviceinfo.DeviceInfoModule"]
  }
}
```

### Step 3 — Android build.gradle

Use `expo-module-gradle-plugin` — it automatically handles SDK versions,
Kotlin setup, and wires `expo-modules-core` as a dependency. No manual
`compileSdk`, no explicit `implementation project(':expo-modules-core')`.

```groovy
// modules/device-info/android/build.gradle
plugins {
  id 'com.android.library'
  id 'expo-module-gradle-plugin'
}

group = 'com.anonymous.myexpoapp'
version = '1.0.0'

android {
  namespace "expo.modules.deviceinfo"
  defaultConfig {
    versionCode 1
    versionName '1.0.0'
  }
}
```

### Step 4 — The Kotlin module

This is where the old and new approaches differ most visibly.

```kotlin
// modules/device-info/android/src/main/.../DeviceInfoModule.kt
package expo.modules.deviceinfo

import android.content.Intent
import android.content.IntentFilter
import android.os.BatteryManager
import android.os.Build
import expo.modules.kotlin.modules.Module
import expo.modules.kotlin.modules.ModuleDefinition

class DeviceInfoModule : Module() {

  override fun definition() = ModuleDefinition {

    // Name — what JS uses in requireNativeModule("DeviceInfo")
    Name("DeviceInfo")

    // Constants — evaluated once at startup, available as sync properties in JS
    Constants {
      mapOf(
        "DEVICE_NAME" to Build.MODEL,
        "OS_VERSION"  to Build.VERSION.RELEASE
      )
    }

    // AsyncFunction — runs on a coroutine, return value goes straight to JS
    // No Promise parameter. Throw an exception to reject.
    AsyncFunction("getBatteryLevel") {
      val filter = IntentFilter(Intent.ACTION_BATTERY_CHANGED)
      val intent = appContext.reactContext?.registerReceiver(null, filter)
        ?: throw Exception("Could not read battery status")

      val level = intent.getIntExtra(BatteryManager.EXTRA_LEVEL, -1)
      val scale = intent.getIntExtra(BatteryManager.EXTRA_SCALE, -1)
      if (level == -1 || scale == -1) throw Exception("Could not read battery level")

      (level.toFloat() / scale.toFloat() * 100).toInt()  // plain Int → JS number
    }

    AsyncFunction("isCharging") {
      val filter = IntentFilter(Intent.ACTION_BATTERY_CHANGED)
      val intent = appContext.reactContext?.registerReceiver(null, filter)
        ?: throw Exception("Could not read battery status")

      val status = intent.getIntExtra(BatteryManager.EXTRA_STATUS, -1)
      status == BatteryManager.BATTERY_STATUS_CHARGING ||
        status == BatteryManager.BATTERY_STATUS_FULL   // plain Boolean → JS boolean
    }

    // Return a Record (data class) — expo-modules-core maps it to a JS object via JSI
    AsyncFunction("getDeviceInfo") {
      DeviceInfoRecord(
        model        = Build.MODEL,
        brand        = Build.BRAND,
        osVersion    = Build.VERSION.RELEASE,
        sdkVersion   = Build.VERSION.SDK_INT,
        deviceId     = Build.ID,
        manufacturer = Build.MANUFACTURER
      )
    }
  }
}
```

### Step 5 — The Record type

Instead of building a `WritableNativeMap` manually, define a data class that
implements `Record`. Each `@Field` property becomes a key in the JS object.

```kotlin
// DeviceInfoRecord.kt
class DeviceInfoRecord : Record {
  @Field var model: String = ""
  @Field var brand: String = ""
  @Field var osVersion: String = ""
  @Field var sdkVersion: Int = 0
  @Field var deviceId: String = ""
  @Field var manufacturer: String = ""

  constructor()
  constructor(
    model: String, brand: String, osVersion: String,
    sdkVersion: Int, deviceId: String, manufacturer: String
  ) { /* assign fields */ }
}
```

Old bridge equivalent (what you no longer need to write):

```kotlin
val map = WritableNativeMap()
map.putString("model", Build.MODEL)
map.putString("brand", Build.BRAND)
// ... repeat for every field
promise.resolve(map)
```

### Step 6 — The config plugin

This is what makes the module survive every `expo prebuild`. It runs during
prebuild and patches the generated `android/` files so Gradle knows about
your local module.

```js
// modules/device-info/plugin/index.js
const { withAppBuildGradle, withSettingsGradle } = require("@expo/config-plugins");

function withDeviceInfoSettings(config) {
  return withSettingsGradle(config, (mod) => {
    if (!mod.modResults.contents.includes("include ':device-info'")) {
      mod.modResults.contents += `
// device-info local Expo Module
include ':device-info'
project(':device-info').projectDir = new File(rootProject.projectDir, '../modules/device-info/android')`;
    }
    return mod;
  });
}

function withDeviceInfoDependency(config) {
  return withAppBuildGradle(config, (mod) => {
    if (!mod.modResults.contents.includes("project(':device-info')")) {
      mod.modResults.contents = mod.modResults.contents.replace(
        `implementation("com.facebook.react:react-android")`,
        `implementation("com.facebook.react:react-android")
    implementation project(':device-info')`
      );
    }
    return mod;
  });
}

module.exports = function withDeviceInfo(config) {
  config = withDeviceInfoSettings(config);
  config = withDeviceInfoDependency(config);
  return config;
};
```

Register it in `app.json`:

```json
"plugins": [
  "expo-router",
  "./modules/device-info/plugin"
]
```

### Step 7 — The TypeScript API

```ts
// modules/device-info/src/index.ts
import { requireNativeModule } from "expo-modules-core";
import { Platform } from "react-native";

const NativeDeviceInfo = (() => {
  try {
    return requireNativeModule("DeviceInfo");
  } catch {
    return null;
  }
})();

const isSupported = Platform.OS === "android" && NativeDeviceInfo !== null;

// Constants are properties directly on the module object
export const constants = {
  DEVICE_NAME: isSupported ? NativeDeviceInfo.DEVICE_NAME as string : "",
  OS_VERSION:  isSupported ? NativeDeviceInfo.OS_VERSION  as string : "",
};

// AsyncFunctions are callable methods on the module object
export async function getBatteryLevel(): Promise<number> {
  return NativeDeviceInfo!.getBatteryLevel();
}

export async function isCharging(): Promise<boolean> {
  return NativeDeviceInfo!.isCharging();
}

export async function getDeviceInfo(): Promise<DeviceInfoData> {
  return NativeDeviceInfo!.getDeviceInfo();
}
```

## Part 4 — Running it

```bash
# 1. Link the local package
npm install

# 2. Apply the config plugin to android/
npx expo prebuild --platform android --no-install

# 3. Build and install on a connected device/emulator
npx expo run:android
```

> **Note:** Expo Go won't work — native modules require a dev build.
> `expo run:android` builds and installs the full APK automatically.

## Part 6 — Old bridge vs Expo Module at a glance

| | Old Bridge | Expo Module |
|---|---|---|
| Base class | `ReactContextBaseJavaModule` | `Module` |
| Registration | Manual `ReactPackage` + `MainApplication` | Auto via `expo-module.config.json` |
| Async methods | `@ReactMethod` + `Promise` param | `AsyncFunction { }` + return value |
| Sync values | `getConstants()` override | `Constants { mapOf(...) }` block |
| Return objects | `WritableNativeMap` | `Record` data class with `@Field` |
| Data transfer | JSON serialisation over bridge | JSI — direct C++ types, zero copy |
| Survives prebuild | ❌ lives in gitignored `android/` | ✅ lives in committed `modules/` |
| New architecture | ❌ bridge only | ✅ JSI native |

## Common pitfalls we hit

**1. Wrong Gradle plugin id**
`expo-module` doesn't exist — the correct id is `expo-module-gradle-plugin`.

**2. Wrong relative path in settings.gradle**
`rootProject.projectDir` resolves from `android/`, so the path to your module
is `../modules/device-info/android`, not `../../modules/...`.

**3. Missing expo-module.config.json**
Without this file, expo-modules-core never generates JSI bindings. Your
functions exist in Kotlin but arrive as `undefined` in JS.

**4. `Constants()` singular vs `Constants {}` block**
`Constant("KEY") { value }` (singular, per-item) and `Constants { mapOf() }`
(block form) are both valid DSL but behave slightly differently on how they
surface in JS. The block form maps reliably to `getConstants()` internals and
is the safer choice.

**5. `package=` in AndroidManifest.xml**
Modern Android Gradle Plugin reads the namespace from `build.gradle`, not the
manifest. Remove `package="..."` from `AndroidManifest.xml` to silence the
warning.
